# 设计系统：运筹 Stratum

## 1. 视觉定位

运筹是一套克制、可信、偏工程工作台气质的 Agent 产品界面。整体信息密度为 Daily App Balanced（6/10），布局差异度为 Balanced Product（5/10），动效强度为 Focused Feedback（4/10）。界面以 Lovable 的暖中性产品语言为基础：奶油色画布、炭黑色文本、同色相透明度层级、细边框和紧凑排版。视觉层级来自留白、字级和边界，不来自装饰性渐变、颗粒、玻璃或厚重阴影。

当前首页 Hero 是未来 Dashboard 的临时占位，保留其内容结构。Chat 工作区是独立的 `Longzhong` 路由；固定导航通过简短的方向反馈在两个页面间切换，不在 Chat 区重复显示工作台标题。

## 2. 配色与职责

颜色分为三个互不混用的层级。

### 产品色

浅色主题：

- **Cream Canvas** `#f7f4ed`：页面画布和默认表面。
- **Charcoal Ink** `#1c1c1c`：主文本、标题和主要操作。
- **Off-white** `#fcfbf8`：深色操作上的文本与高对比强调。
- **Muted Gray** `#5f5f5d`：辅助文案、占位符和时间。
- **Passive Border** `#eceae4`：静态边框与分隔线。
- **Interactive Border** `rgba(28, 28, 28, 0.4)`：输入和可交互边界。
- **Focus Ring** `rgba(59, 130, 246, 0.5)`：键盘焦点。

暗色主题反转奶油色与炭黑色：`#1c1c1c` 为画布，`#fcfbf8` 为主文本，表面、辅助文本和边框由奶油色的 4%–40% 透明度派生。不得重新引入蓝灰色或紫灰色产品表面。

### 语义色

成功、警告、危险和信息色只表达运行状态、审批或错误。语义色不得成为装饰性强调，也不得替代产品操作色。正文与状态文案必须满足实际表面的对比度要求。

### Stratum 特效色

既有 Cloud Dancer、Baltic Sea、Blue Fusion、Quiet Violet 等多色值仅保留给品牌标识和未来有明确沟通目的的特效。本阶段不得用于页面背景、导航、输入区、普通卡片、正文或主要控件，也不得新增环境式特效。

## 3. 排版

- **展示与中文：** `Noto Sans Variable`。
- **正文与界面：** `Nunito Sans Variable`；正文最大宽度 `65ch`。
- **字重：** 正常层级只使用 400 与 600。
- **对话正文：** 桌面和移动端均不低于 16px。

| 角色               | 字号                              | 字重 | 行高 | 字距                    |
| ------------------ | --------------------------------- | ---- | ---- | ----------------------- |
| Hero               | 桌面 60px / 平板 48px / 移动 36px | 600  | 1.05 | 桌面 -1.5px，按比例缩放 |
| 章节标题           | 48px                              | 600  | 1.00 | -1.2px                  |
| 次级标题           | 36px                              | 600  | 1.10 | -0.9px                  |
| 卡片标题           | 20px                              | 400  | 1.25 | 0                       |
| 大号正文           | 18px                              | 400  | 1.38 | 0                       |
| 正文与标准操作     | 16px                              | 400  | 1.50 | 0                       |
| 说明文字与紧凑控件 | 14px                              | 400  | 1.50 | 0                       |

中英文使用相同语义层级。紧凑操作可使用 14px，但不得通过缩小字号解决桌面操作文案换行；应优先调整可用宽度或截断策略。

## 4. 表面、形状与组件

- **平面内容：** 无外框、无阴影，以间距组织。
- **静态容器：** 1px Passive Border，无投影。
- **浮层：** 可使用极轻的 `0 12px 32px rgba(28, 28, 28, 0.12)` 纵深。
- **控件：** 6px 圆角。
- **标准容器、输入区、卡片：** 12px 圆角。
- **大型浮层：** 最大 16px 圆角。
- **全圆角：** 仅用于圆形图标操作、开关或形状确实表达行为的控件。
- **按钮：** 主操作使用炭黑底与米白字，次操作使用透明底和交互边框；按下只下移 1px，触控目标至少 44px。
- **导航：** 两个路由使用相同奶油表面、细边框、6px 控件语言；当前项由文字加深和 2px 短线表达，不使用胶囊底色。
- **Composer：** 单层 12px 容器，最大宽度约 896px，常态最小高度约 116px。输入正文位于上层，Agent、Model、状态操作位于约 44px 底栏，发送按钮保持 44px 圆形操作。不得卡片套卡片。
- **消息：** 助手消息直接落在页面背景；用户消息仅使用轻微产品表面区分。推理和工具默认折叠，语义状态始终可辨认。

受保护目录 `app/components/ui`、`app/components/react-bits` 和 `app/components/ai-elements` 的内部实现不得修改；通过语义 token、props、Stratum 自有包装层和使用方样式适配。

## 5. 布局

### 路由

1. **概览 `/`：** 当前 Hero / 未来 Dashboard 占位，`min-height: 100dvh`。
2. **隆中对 `/longzhong`：** 独立的真实 Agent 对话工作区，为固定导航与 Composer 保留安全空间。

两个页面是独立路由，不放进共享纵向滚动轨道。

### 导航

- 左侧品牌为“运筹”。
- 右侧顺序为“概览、隆中对、分隔线、语言、主题、注册”。
- 两个路由可使用不同最大宽度，但表面、边框、排版和形状必须一致。
- 导航选中态已经表达页面位置，Chat 区不重复标题。

### Chat 工作区

- 始终是居中单列；桌面端只调整左右 gutter，不增加永久侧栏或空事件列。
- 历史会话为可切换 overlay / drawer，不进入主布局流。
- Composer 与导航共享最大约 896px 外宽；对话正文保持约 800px 阅读宽度。
- 消息使用 document scroll；固定 Composer 不得遮挡最后一条消息。
- Navbar 与 Composer 的视口外边距由各自最外层 fixed container 表达。
- Agent、Model、事件投影、审批、恢复和历史数据全部保持真实 API 或本地持久化事实，不得伪造。

### 响应式

- `< 768px` 为单列，无横向滚动。
- Hero 字号依次为 36px、48px、60px。
- 页面 gutter 最小 12px；Composer 使用视口减 24px 的可用宽度。
- Agent / Model 触发器可收缩并截断；发送、历史、语言、主题和注册仍保持至少 44px 触控目标。
- 历史记录仍为浮层，输入区保持可见，最后一条消息不被遮挡。

## 6. 动效与交互

- Stratum 自有的悬停、聚焦、选择、开合与路由反馈控制在 150–250ms。
- 已有编排需求继续使用 GSAP；简单状态使用 CSS。优先只动画 `transform` 与 `opacity`。
- 路由切换保留方向感，但不展示长入场序列。
- 导航指示器只动画 `transform` 与 `scaleX`，不得动画 `left`、`width`、`top` 或 `height`。
- 历史浮层使用短促的 `opacity + translate`，关闭时恢复触发器焦点。
- 自动跟随仅在用户位于真实底部时启用；用户向上阅读后保持位置，点击“滚动到底部”才恢复。
- `prefers-reduced-motion: reduce` 下所有路由、浮层、选择和工作区过渡立即完成，内容与状态仍完整可见。
- 不增加环境式、永久循环、滚动驱动或纯装饰动画。

## 7. 禁止项

- 不使用 emoji、Inter、通用衬线字体或纯黑。
- 不使用紫色/蓝色霓虹、外发光、高饱和渐变、全局颗粒或玻璃效果。
- 不使用普通卡片厚阴影，不把中央对话流包进大卡片，不做卡片套卡片。
- 不把 history 变成永久 rail，不创建空的事件列，不使用三等分卡片布局。
- 不重做当前 Hero 的内容结构。
- 不伪造历史、工具状态、审批解释或后端成功状态。
- 不用绝对定位堆叠正文内容；每个内容区域必须保留清晰文档流。
