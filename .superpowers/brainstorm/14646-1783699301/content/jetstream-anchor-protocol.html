JetStream anchor protocol

The checkpointed AgentEvent::Message { seq, ... } is the anchor. No extra commit event.

Agent-scoped subjects
wyse.events.agent.<agent_id>.message.<seq>
wyse.events.agent.<agent_id>.run.<run_id>.realtime

Subscribe to wyse.events.agent.<agent_id>.> for the whole Agent lifetime. The exact message subject lets the server resolve business seq to JetStream's internal stream sequence.

last_seq = N
messages/N.json stores the exact seq-bearing envelope.
Publish it to the exact message subject with deterministic Nats-Msg-Id = agent_id:N.
Only after JetStream ack, CAS-advance agent.json.last_seq.
Invariant: if agent.json.last_seq = N, both the checkpoint file and its JetStream anchor have existed.
A live subscriber can briefly see message N before last_seq advances. The seq-bearing event is already staged in the filesystem; another writer can help finish its CAS commit.

Client recovery from front_seq = F

Read a fixed barrier agent.json.last_seq = L.
Direct-get subject ...message.L; extract JetStream stream sequence J.
Immediately create an ordered Agent consumer at J + 1; buffer its events.
Page stable checkpoint history (F, L] from messages/.
After history is applied, drain buffered JetStream events, then keep the same consumer open for live traffic.
Messages committed concurrently after barrier L arrive through JetStream, so they are not duplicated in the fixed history page and are not missed.
If anchor L has expired under retention, stable recovery still succeeds from files, but transient replay cannot be exact. Return a reset/gap signal and begin from live traffic.
Why publish cannot remain warning-only for seq-bearing events: advancing last_seq before JetStream acknowledges the same event can leave a committed checkpoint with no stream anchor. Realtime-only events may remain best-effort; checkpoint message events may not.
