Decoupled JetStream + concurrent send handling

JetStream durability is useful, but it is not part of the checkpoint commit. Agent CAS independently prevents two processes from starting overlapping LLM runs.

Two independent durable planes

Filesystem
agent.json and messages/{seq}.json are the authoritative stable history.
JetStream
File-backed retained runtime log. It persists every event that NATS accepted, including deltas.
1
Commit the stable message using filesystem CAS only.
2
Publish the same seq-bearing message event to JetStream independently.
3
If publish fails, keep the checkpoint; refresh recovers stable history. No rollback and no anchor requirement.
A client with a JetStream consumer cursor can resume retained events. A hard refresh can load checkpoint history and begin a fresh live consumer; losing a rare in-flight draft is acceptable.

Two users send to the same Agent

status = idlelast_seq = N
1
A and B both read the same idle Agent version and both target N + 1.
2
Only A wins put(messages/N+1.json, Absent). B cannot overwrite it.
3
A CAS-updates agent.json in one write: last_seq=N+1, status=running, and A's run_id/turn_id.
4
B re-reads after the conflict, sees status=running, does not persist its own message and does not start another stream. It returns Agent-busy/retry.
5
Assistant/tool appends must match the active run_id/turn_id; stale writers fail closed.
Result: one Agent has at most one active LLM stream across processes. No interleaved histories and no speculative queue.
If the winning process crashes after the running state is committed, ordinary sends still see busy. Explicit runtime recovery must resume that persisted run; last_seq alone cannot distinguish “alive” from “crashed.”
